# 贪吃蛇项目协作约定

本文件用于固化当前仓库的开发协作规则。后续在本项目中进行分析、设计、编码、重构、讲解、阶段汇报和提交时，默认遵循以下要求。

## 1. 项目目标

使用 Python + Pygame 开发一个具备良好扩展性的贪吃蛇客户端游戏，并按工程化方式分阶段推进。

## 2. 开发流程要求

1. 先输出简洁但专业的需求分析文档。
2. 再输出项目架构设计，包括：
   - 项目目录结构
   - 项目架构设计
   - 页面设计
   - UI 设计
   - 模块划分
   - 类设计
   - 数据流设计
3. 然后将项目拆分为多个迭代版本：
   - MVP
   - 菜单
   - 设置
   - 存档
   - 优化
4. 每次只完成一个小阶段，不一次性输出全部代码。
5. 每个阶段都要先说明目标、实现思路、涉及文件，然后再进入代码修改。

## 3. 阶段报告固定格式

每完成一个阶段、一个 Sprint 或一个明确的小修复后，必须按下面格式汇报。汇报要具体到页面、UI、按钮、操作结果、数据流和业务完成进度，不能只写“完成了某功能”。

```markdown
## 阶段 X：阶段名称

当前阶段目标：
- 写清楚本阶段要解决的问题。
- 写清楚本阶段边界，不属于本阶段的内容也要说明。

本阶段完成功能：
- 逐条列出用户能感知到的功能变化。
- 如果涉及键盘，说明具体按键和结果。
- 如果涉及鼠标，说明点击或悬停的结果。
- 如果涉及存档、设置、分数等数据，说明数据是否保存、读取或更新。

本阶段页面与 UI 完成情况：

页面名 `SceneName`：
- 页面标题：说明标题文字和渲染组件。
- UI 组件：说明本页面用了哪些组件，例如 `TextBlock`、`MenuList`、`Button`。
- 按钮或菜单项：逐个说明能不能点、如何选中、触发后调用什么逻辑。
- 键盘操作：逐个说明按键和响应结果。
- 鼠标操作：逐个说明点击、悬停和响应结果。
- 状态提示：说明页面上会显示哪些状态文案。

本阶段完成设计：
- 说明新增或调整了哪些模块、类、组件。
- 页面设计和 UI 设计必须一一对应。
- 说明场景层、UI 层、模型层、服务层的边界是否变化。
- 说明本阶段为什么这样拆分，后续如何扩展。

本阶段数据流设计：
- 逐条写出关键数据流。
- 格式示例：`A 页面接收输入 -> 调用 B 服务 -> 更新 C 数据 -> 切换到 D 页面`。
- 如果涉及 JSON，必须说明读写哪个 JSON 文件。
- 如果涉及存档，必须说明创建、更新、读取、删除分别在哪里发生。

本阶段核心业务场景完成进度：
- 主菜单业务场景：说明完成到哪里。
- 设置业务场景：说明完成到哪里。
- 游戏业务场景：说明完成到哪里。
- 存档业务场景：说明完成到哪里。
- UI 组件化业务场景：说明完成到哪里。
- 输入业务场景：说明完成到哪里。
- 未完成内容：说明留到后续哪个阶段。

涉及文件：
- `path/to/file.py`
- `path/to/doc.md`

验证情况：
- 写明执行过的语法检查、自动验证、人工验证。
- 写明哪些验证因为环境限制只能由用户手动确认。
- 写明是否已提交并推送到 GitHub。
- 如果已提交，写明 commit hash 和 commit message。

下一阶段建议：
- 写明下一阶段建议继续做什么。
- 如果有多个可选方向，按优先级列出。
```

## 4. 架构图格式要求

后续输出项目架构设计时，架构图必须使用分层框图格式，例如：

```text
┌─────────────────────────────────────────────────────┐
│                   UI 展示层                          │
│  菜单页面 / 设置页面 / 游戏页面 / 继续游戏页面          │
├─────────────────────────────────────────────────────┤
│                  应用控制层                           │
│  App / 场景切换 / 主循环 / Pygame 生命周期             │
├─────────────────────────────────────────────────────┤
│                  场景业务层                           │
│  MenuScene / GameplayScene / SettingsScene           │
├─────────────────────────────────────────────────────┤
│                  领域模型层                           │
│  Snake / Food / GameState                            │
├─────────────────────────────────────────────────────┤
│                  输入服务层                           │
```

不要只用简单文字列表替代架构图。

## 5. 代码实现要求

1. 所有代码必须保证可运行。
2. 代码结构必须清晰，便于扩展。
3. 禁止把所有逻辑都写进一个文件。
4. 优先采用面向对象设计。
5. 添加必要注释，但不要过度冗余。
6. 配置和存档统一使用 JSON 文件。
7. 编码时必须考虑：
   - 异常处理
   - 边界情况
   - 后续扩展能力

## 6. 功能迭代顺序

严格按以下顺序推进：

1. 先做最小可运行版本：
   - 蛇移动
   - 食物
   - 计分
   - 游戏结束
2. 然后逐步加入：
   - 主菜单
   - 设置
   - 存档
   - 工程优化

## 7. 交互规则

1. 用户说“继续下一步”或“继续开发”时，再进入下一个阶段。
2. 用户说“重构”时，按工程化方式优化当前代码结构。
3. 用户说“解释”时，用适合初学者理解的方式讲解代码。
4. 每完成一个阶段后，都需要按“阶段报告固定格式”汇报，并明确下一阶段建议。
5. 每次 Git commit 后，必须说明：
   - 当前 Sprint 是哪个阶段
   - 本次 commit hash 和 message
   - 下一个 Sprint 是哪个阶段

## 8. GitHub 提交规则

1. 每完成一个阶段或一个明确的小修复，必须提交到 GitHub。
2. 提交前必须先做必要验证。
3. 不要把运行时脏数据误提交为功能代码。
4. 如果 `data/save.json` 因启动游戏产生临时存档，要确认是否属于本阶段目标；如果不是，恢复为空存档。

## 9. 当前默认执行策略

1. 默认先按文档和架构设计推进，再进入编码。
2. 编码时优先保持模块边界清晰。
3. 每次修改尽量控制在当前阶段目标内，避免提前引入后续阶段复杂度。
4. 若需求与当前阶段边界冲突，优先提醒并建议放入后续迭代。
5. 如果用户指出流程或规范遗漏，应优先修正 `AGENTS.md`，再继续后续开发。
