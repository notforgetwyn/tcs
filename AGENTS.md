# 贪吃蛇项目协作约定

本文件用于固化当前仓库的开发协作规则。后续在本项目中进行分析、设计、编码、重构、讲解、阶段汇报和提交时，默认遵循以下要求。

## 1. 项目目标

使用 Python + Pygame 开发一个具备良好扩展性的贪吃蛇客户端游戏，并按工程化方式分阶段推进。

## 2. 开发流程要求

1. 先输出简洁但专业的需求分析文档。
2. 再输出项目架构设计，包括：
   - 项目目录结构
   - 项目架构设计
   - 页面设计
   - UI 设计
   - 模块划分
   - 类设计
   - 数据流设计
3. 然后将项目拆分为多个迭代版本：
   - MVP
   - 菜单
   - 设置
   - 存档
   - 优化
4. 每次只完成一个小阶段，不一次性输出全部代码。
5. 每个阶段都要先说明：
   - 目标
   - 实现思路
   - 涉及文件
   然后再进入代码修改。

## 3. 阶段报告固定格式

每完成一个阶段或一个 Sprint 补充修复后，必须按以下格式汇报，不允许只简单说“完成了”。

```markdown
### 阶段 X：阶段名称

**当前阶段目标**：
- 说明本阶段要解决什么问题，本阶段边界是什么。

**本阶段完成功能**：
- 逐条列出用户能感知到的功能变化。
- 如果涉及页面交互，要说明哪个按钮能不能点、点了有什么反应。
- 如果涉及键盘或鼠标，要说明具体按键/操作和结果。

**本阶段完成设计**：
- 说明本阶段新增或调整了哪些模块、类、组件、数据流。
- 页面设计和 UI 设计必须一一对应。
- 对每个页面说明：
  - 页面包含哪些 UI 组件
  - 组件状态有哪些
  - 用户操作后如何流转
- 对核心业务说明：
  - 当前业务流程完成到了哪里
  - 数据从哪里来
  - 数据到哪里去
  - 哪些逻辑留到后续阶段

**涉及文件**：
- `path/to/file.py`
- `path/to/doc.md`

**验证情况**：
- 已执行哪些检查。
- 哪些人工验证已完成。
- 哪些验证因为环境限制未完成。

**GitHub 提交情况**：
- 当前 Sprint：`Sprint X：名称`
- 本次提交：`commit_hash commit message`
- 是否已推送到 GitHub。

**下一阶段建议**：
- 说明下一阶段建议做什么。
```

## 4. 架构图格式要求

后续输出项目架构设计时，架构图必须使用分层框图格式，例如：

```text
┌─────────────────────────────────────────────────────┐
│                   UI 展示层                          │
│  菜单页面 / 设置页面 / 游戏页面 / 继续游戏页面          │
├─────────────────────────────────────────────────────┤
│                  应用控制层                           │
│  App / 场景切换 / 主循环 / Pygame 生命周期             │
├─────────────────────────────────────────────────────┤
│                  场景业务层                           │
│  MenuScene / GameplayScene / SettingsScene           │
```

## 5. 代码实现要求

1. 所有代码必须保证可运行。
2. 代码结构必须清晰，便于扩展。
3. 禁止把所有逻辑都写进一个文件。
4. 优先采用面向对象设计。
5. 添加必要注释，但不要过度冗余。
6. 配置和存档统一使用 JSON 文件。
7. 编码时必须考虑：
   - 异常处理
   - 边界情况
   - 后续扩展能力

## 6. 功能迭代顺序

严格按以下顺序推进：

1. 先做最小可运行版本：
   - 蛇移动
   - 食物
   - 计分
   - 游戏结束
2. 然后逐步加入：
   - 主菜单
   - 设置
   - 存档
   - 工程优化

## 7. 交互规则

1. 用户说“继续下一步”或“继续开发”时，再进入下一个阶段。
2. 用户说“重构”时，按工程化方式优化当前代码结构。
3. 用户说“解释”时，用适合初学者理解的方式讲解代码。
4. 每完成一个阶段后，都需要明确说明下一阶段建议做什么。
5. 每次 Git commit 后，必须说明：
   - 当前 Sprint 是哪个阶段
   - 本次 commit hash 和 message
   - 下一个 Sprint 是哪个阶段

## 8. GitHub 提交规则

1. 每完成一个阶段或一个明确的小修复，必须提交到 GitHub。
2. 提交前必须先做必要验证。
3. 不要把运行时脏数据误提交为功能代码。
4. 如果 `data/save.json` 因启动游戏产生临时存档，要确认是否属于本阶段目标；如果不是，恢复为空存档。

## 9. 当前默认执行策略

1. 默认先按文档和架构设计推进，再进入编码。
2. 编码时优先保持模块边界清晰。
3. 每次修改尽量控制在当前阶段目标内，避免提前引入后续阶段复杂度。
4. 若需求与当前阶段边界冲突，优先提醒并建议放入后续迭代。
5. 如果用户指出流程或规范遗漏，应优先修正 `AGENTS.md`，再继续后续开发。
