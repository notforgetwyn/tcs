# 贪吃蛇项目协作约定

本文件用于固化当前仓库的开发协作规则。后续在本项目中进行分析、设计、编码、重构和讲解时，默认遵循以下要求。

## 1. 项目目标

使用 Python + Pygame 开发一个具备良好扩展性的贪吃蛇客户端游戏，并按工程化方式分阶段推进。

## 2. 开发流程要求

1. 先输出简洁但专业的需求分析文档。
2. 再输出项目架构设计，包括：
   - 项目目录结构
   - 项目架构设计
   - 页面设计
   - UI 设计
   - 模块划分
   - 类设计
   - 数据流设计
3. 然后将项目拆分为多个迭代版本：
   - MVP
   - 菜单
   - 设置
   - 存档
   - 优化
4. 每次只完成一个小阶段，不一次性输出全部代码。
5. 每个阶段都要先说明：
   - 目标
   - 实现思路
   - 涉及文件
   然后再给代码。

## 3. 代码实现要求

1. 所有代码必须保证可运行。
2. 代码结构必须清晰，便于扩展。
3. 禁止把所有逻辑都写进一个文件。
4. 优先采用面向对象设计。
5. 添加必要注释，但不要过度冗余。
6. 配置和存档统一使用 JSON 文件。
7. 编码时必须考虑：
   - 异常处理
   - 边界情况
   - 后续扩展能力

## 4. 功能迭代顺序

严格按以下顺序推进：

1. 先做最小可运行版本：
   - 蛇移动
   - 食物
   - 计分
   - 游戏结束
2. 然后逐步加入：
   - 主菜单
   - 设置
   - 存档

## 5. 交互规则

1. 用户说“继续下一步”时，再进入下一个阶段。
2. 用户说“重构”时，按工程化方式优化当前代码结构。
3. 用户说“解释”时，用适合初学者理解的方式讲解代码。
4. 每完成一个阶段后，都需要明确说明下一阶段建议做什么。

## 6. 当前默认执行策略

1. 默认先按文档和架构设计推进，再进入编码。
2. 编码时优先保持模块边界清晰。
3. 每次修改尽量控制在当前阶段目标内，避免提前引入后续阶段复杂度。
4. 若需求与当前阶段边界冲突，优先提醒并建议放入后续迭代。
