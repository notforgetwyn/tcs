# Python + Pygame 贪吃蛇项目蓝图

## 1. 协作规则

本项目按以下规则推进：

1. 先输出简洁但专业的需求分析文档。
2. 再输出项目架构设计，包括项目目录结构、项目架构设计、页面设计、UI 设计、模块划分、类设计、数据流设计。
3. 然后把项目拆成多个迭代版本：MVP、菜单、设置、存档、优化。
4. 每次只完成一个小阶段，不一次性写完整个项目。
5. 每个阶段都先说明目标、实现思路、涉及文件，再给代码。
6. 所有代码必须可运行、结构清晰、便于扩展，禁止把所有逻辑写进一个文件。
7. 优先使用面向对象设计，加入必要注释，但不过度冗余。
8. 配置和存档统一使用 JSON 文件。
9. 写代码时考虑异常处理、边界情况和后续扩展。
10. 每完成一个阶段，都给出下一阶段建议。

额外约束：

- 先从最小可运行版本开始：蛇移动、食物、计分、游戏结束。
- 然后再逐步加入主菜单、设置、存档。
- 用户说“继续下一步”后，才进入下一个阶段。
- 用户说“重构”后，按工程化方式优化当前结构。
- 用户说“解释”后，用适合初学者的方式解释代码。

## 2. 需求分析

### 2.1 项目目标

开发一个基于 Python + Pygame 的桌面端贪吃蛇游戏，具备可玩性、可维护性和可扩展性，并按照工程化迭代方式逐步建设。

### 2.2 核心功能需求

1. 贪吃蛇支持上、下、左、右移动。
2. 蛇吃到黄色食物后增加分数。
3. 游戏过程中实时显示分数。
4. 游戏启动后先进入开始界面，而不是直接开始。
5. 开始界面支持：
   - 开始游戏
   - 继续游戏
   - 设置
   - 退出游戏
6. 支持游戏存档与读档。
7. 设置页面支持调整蛇移动速度，并持久化保存。
8. 整体架构支持后续扩展，例如暂停、皮肤、音效、障碍物、排行榜等功能。

### 2.3 非功能需求

1. 代码结构清晰，模块职责明确。
2. 采用面向对象设计。
3. 配置与存档使用 JSON 存储。
4. 具备基本异常处理能力，例如配置文件缺失、存档损坏、非法数据回退默认值。
5. 便于测试和后续重构。

### 2.4 边界与风险

1. 输入方向切换时，不能直接反向撞向自己。
2. 食物生成不能落在蛇身体上。
3. 配置文件或存档文件不存在时，需要自动创建或回退默认值。
4. 读档时需要校验数据合法性，避免程序崩溃。
5. 游戏速度调整需要设置合理上下限，避免不可玩。

#### 分层说明

1. UI 展示层
   - 负责页面展示、按钮交互、文本显示和 HUD 渲染。
   - 不直接持久化数据，不承担底层文件读写职责。

2. 应用控制层
   - 负责程序启动、Pygame 初始化、主循环管理、事件分发和场景切换。
   - 连接 UI 展示层与业务逻辑层，是运行时调度中心。

3. 业务逻辑层
   - 负责蛇移动、食物生成、计分、碰撞检测、胜负判定等核心规则。
   - 尽量不依赖具体页面，以保证后续可测试和可复用。

4. 配置与存档服务层
   - 负责设置和存档的读取、保存、校验、默认值补全和异常兜底。
   - 向上为场景和应用层提供统一的数据访问接口。

5. 文件管理层
   - 负责组织配置目录、存档目录、资源目录和文档目录。
   - 屏蔽具体文件路径，避免业务代码散落硬编码路径。

6. 数据存储层
   - 当前阶段采用 JSON，适合设置、存档和轻量数据持久化。
   - 后续如果出现排行榜、成就或复杂用户数据，再考虑升级到 SQLite。

7. 日志与诊断预留层
   - 当前版本先预留接口和位置。
   - 后续可加入日志记录、错误诊断和调试开关，提升可维护性。

#### 架构设计原则

1. UI 与业务逻辑解耦，避免把游戏规则直接写死在页面渲染代码中。
2. 配置与存档统一通过服务层访问，避免后续功能增加时重复读写 JSON。
3. 主循环只负责调度，不承载过多业务细节。
4. 各层职责单一，便于后续加入暂停页、音效系统、排行榜和主题皮肤。

页面目标：
- 作为程序统一入口

- 顶部标题：`Snake Game`
- 中部菜单项：

  - 继续游戏
  - 设置
  - 退出游戏

- `Up/Down` 或 `W/S` 切换选项

- 背景：深灰/黑色
- 标题：白色大字号

- 选中菜单项：蓝色高亮

页面目标：
- 承载核心玩法

### 3.4 全局 UI 规范

为保证每个页面风格统一，定义以下通用规范：

1. 颜色规范
   - 背景：深灰/黑色
   - 主文本：白色
   - 强调色：蓝色
   - 警告/结束：红色
   - 成功/主角色：绿色
   - 食物：黄色
2. 布局规范
   - 页面主内容尽量居中
   - HUD 固定角落显示，不遮挡核心玩法
   - 菜单类页面采用垂直排列
3. 组件规范
   - 文本组件：标题、正文、提示三种字号
   - 按钮组件：默认态、选中态、禁用态
   - 提示组件：用于无存档、保存成功、错误信息
4. 扩展预留
   - 后续可加入图标、动效、音效和主题皮肤

### 3.7 数据流设计

```text
用户输入
   ↓
当前场景接收事件
   ↓
应用控制层分发动作
   ↓
业务逻辑层更新游戏状态
   ↓
配置/存档服务层按需读写 JSON
   ↓
场景重新渲染 UI
```

#### 运行时数据流

```text
键盘输入
   ↓
GameplayScene.handle_event()
   ↓
Snake.set_direction()
   ↓
App 主循环触发 update()
   ↓
Snake.move()
   ↓
碰撞检测 / 吃食物判定 / 分数更新
   ↓
GameplayScene.render()
   ↓
屏幕显示最新游戏状态
```

#### 配置数据流

```text
程序启动
   ↓
SettingsService.load_settings()
   ↓
读取 config/settings.json
   ↓
非法或缺失则回退默认值
   ↓
应用到游戏运行参数
   ↓
用户在设置页修改速度
   ↓
SettingsService.save_settings()
   ↓
写回 settings.json
```

#### 存档数据流

```text
游戏运行中
   ↓
GameSession / GameData 生成当前状态
   ↓
SaveService.save_game()
   ↓
写入 data/save.json
   ↓
主菜单点击继续游戏
   ↓
SaveService.load_save()
   ↓
恢复蛇位置、方向、分数、速度等状态
   ↓
GameplayScene 根据存档恢复运行
```

#### 数据流设计原则

1. 页面层只消费数据和触发动作，不直接操作底层文件。
2. 业务状态变化先发生在内存对象，再通过服务层决定是否持久化。
3. JSON 文件始终通过统一入口读写，避免多个模块各自维护格式。
4. 数据流方向尽量单向，降低后续调试和扩展成本。

## 4. 迭代规划

### Iteration 1: MVP

目标：
- 实现最小可运行版本
- 包含蛇移动、食物、计分、游戏结束

范围：
- 暂不做主菜单
- 暂不做设置页

## 5. 第一阶段开发计划

### 5.1 阶段目标

完成一个最小可运行的单局游戏版本，具体包括：

1. 玩家可以使用方向键控制蛇移动
2. 蛇可以吃到黄色食物
3. 分数实时增长并显示
4. 撞墙或撞自己后游戏结束

### 5.2 实现思路

第一阶段不引入完整场景系统，但代码结构仍按后续可扩展方式组织：

1. 先建立基础项目目录
2. 搭建 `main.py + app.py + models + constants`
3. 先只实现 `GameplayScene` 所需最小集合
4. 使用网格坐标而不是像素坐标管理蛇和食物，简化逻辑
5. 使用 Pygame `Clock` 控制刷新率，使用“定时步进”控制蛇速度
